# Portsie Adaptive Ingestion Architecture Review

## Context and Objective

This document reviews the current Portsie architecture and proposes a rigorous, implementation-ready plan to support:

1. mixed ingestion from APIs and uploaded documents in ad hoc user flows,
2. reliable source/type detection for any new upload,
3. controlled schema evolution so new data structures can be absorbed safely over time.

Primary goal: every new source should be easier to onboard than the last one, while preserving strict data integrity and privacy.

---

## Current Architecture (What Is Already Strong)

### Existing strengths in the codebase

- `upload` pipeline is already split into deterministic stages:
  - Stage 1+2: `POST /api/upload/[id]/extract`
  - Stage 2.5+3: `POST /api/upload/[id]/confirm`
  - Verification: `POST /api/upload/[id]/verify`
- Extraction schema (`PortsieExtraction v1`) is explicit, versioned, and validated.
- Deterministic account matching and deterministic DB writer are already implemented.
- Canonical financial tables already exist and are reused across sources:
  - `accounts`, `transactions`, `position_snapshots`, `balance_snapshots`, `holdings`
- Multiple providers already coexist (`schwab_api`, `manual_upload`, `quiltt`, etc.).
- Reliability and diagnostics are improving (`processing_log`, `write_warnings`, `extraction_failures` metadata).
- Privacy hardening already removed raw LLM responses and introduced redacted debug context.

### Current architectural tension

The system is functionally correct but source onboarding and schema evolution are still partly "code-first/manual" rather than "data-model-governed":

- adding a source often means editing several CHECK constraints + TS enums,
- unknown fields from new documents/APIs do not have a first-class "landing zone",
- schema changes can be proposed informally without a strict acceptance gate.

---

## Target Architecture

Keep current canonical tables as the "golden core", but add an explicit **adaptive ingestion layer** before canonical writes.

### Four-layer ingestion model

1. **Capture layer**
   - store source payload metadata and detection output
   - never lose unknown fields
2. **Normalization layer**
   - map source-specific structures into a canonical extraction envelope
3. **Governance layer**
   - track unknown fields, frequency, and confidence
   - produce schema-change proposals (not auto-migrations)
4. **Canonical write layer**
   - write only validated canonical data into core financial tables

---

## Design Principles (Strict)

1. **No direct schema mutation from runtime**
   - runtime can suggest, never alter schema.
2. **Unknown data is stored, not discarded**
   - unknown field/value shapes go to dedicated JSONB tables.
3. **Canonical tables remain stable**
   - avoid adding source-specific columns to core tables.
4. **Schema upgrades are versioned and audited**
   - proposal -> approval -> migration -> rollout.
5. **Source onboarding should be mostly configuration + adapter mapping**
   - not repetitive constraint surgery.

---

## Proposed Data Model Additions

Add these tables (or equivalent names) via migration.

### 1) `ingestion_sources`
- registry of logical sources (`schwab`, `quiltt`, `manual_upload_pdf`, future connectors)
- metadata: source type (`api`, `file`, `hybrid`), activation, owner

### 2) `ingestion_runs`
- one record per ingestion attempt (upload extract, API sync, webhook ingest)
- track status, durations, source, model/backend, user/account references

### 3) `source_schema_registry`
- stores observed structure signatures and canonical schema version compatibility
- examples: hash of JSON keys, CSV header signature, OFX tag set signature

### 4) `ingestion_observations`
- persistent JSONB landing zone for unknown/unmapped fields
- includes path (`accounts[0].balances[0].foo`) and sample values

### 5) `schema_change_proposals`
- generated by analysis jobs, reviewed by humans/agents
- fields: proposal type, evidence count, affected source signatures, risk score

## Suggested SQL Migration Example

```sql
-- 1) Source registry
create table if not exists ingestion_sources (
  id uuid primary key default gen_random_uuid(),
  key text not null unique,
  source_type text not null check (source_type in ('api', 'file', 'hybrid')),
  enabled boolean not null default true,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- 2) Per-run execution tracking
create table if not exists ingestion_runs (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  source_id uuid not null references ingestion_sources(id),
  uploaded_statement_id uuid references uploaded_statements(id) on delete set null,
  run_kind text not null check (run_kind in ('extract', 'confirm', 'verify', 'api_sync', 'webhook')),
  status text not null check (status in ('pending', 'running', 'completed', 'partial', 'failed')),
  backend text,
  model text,
  started_at timestamptz not null default now(),
  finished_at timestamptz,
  duration_ms integer,
  error_category text,
  error_message text,
  diagnostics jsonb not null default '{}'::jsonb
);
create index if not exists idx_ingestion_runs_user_started on ingestion_runs(user_id, started_at desc);

-- 3) Observed source structure signatures
create table if not exists source_schema_registry (
  id uuid primary key default gen_random_uuid(),
  source_id uuid not null references ingestion_sources(id),
  structure_signature text not null,
  schema_version integer not null,
  first_seen_at timestamptz not null default now(),
  last_seen_at timestamptz not null default now(),
  seen_count integer not null default 1,
  sample_payload jsonb,
  unique (source_id, structure_signature, schema_version)
);

-- 4) Unknown/unmapped observations
create table if not exists ingestion_observations (
  id uuid primary key default gen_random_uuid(),
  ingestion_run_id uuid not null references ingestion_runs(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  source_id uuid not null references ingestion_sources(id),
  field_path text not null,
  observed_type text,
  observed_value jsonb,
  confidence text check (confidence in ('high', 'medium', 'low')),
  created_at timestamptz not null default now()
);
create index if not exists idx_ingestion_observations_path on ingestion_observations(source_id, field_path);

-- 5) Controlled schema proposals
create table if not exists schema_change_proposals (
  id uuid primary key default gen_random_uuid(),
  proposal_key text not null unique,
  status text not null check (status in ('open', 'approved', 'rejected', 'implemented')),
  proposal_type text not null check (proposal_type in ('enum_extension', 'new_field', 'new_table', 'constraint_change')),
  rationale text not null,
  evidence_count integer not null default 0,
  risk_score integer not null default 0,
  proposal_payload jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now(),
  decided_at timestamptz
);
```

## Source and File-Type Detection Strategy

Detection should not rely only on MIME type. Use a deterministic multi-signal classifier:

1. file metadata (MIME, extension, size),
2. magic bytes / header markers (`%PDF`, OFX tags, JSON object start, CSV delimiters),
3. structural profile (CSV headers, JSON top-level keys, OFX sections),
4. optional LLM-assisted fallback classification only when deterministic confidence is low.

### TypeScript detector example

```ts
export type DetectedSourceKind =
  | "brokerage_statement_pdf"
  | "transactions_csv"
  | "positions_csv"
  | "ofx_qfx"
  | "api_json_export"
  | "unknown";

export interface DetectionResult {
  kind: DetectedSourceKind;
  confidence: "high" | "medium" | "low";
  reasons: string[];
  structureSignature: string;
}

export function detectUploadKind(
  filename: string,
  mimeType: string,
  buffer: Buffer
): DetectionResult {
  const textPrefix = buffer.toString("utf8", 0, Math.min(buffer.length, 4096));
  const reasons: string[] = [];

  if (textPrefix.startsWith("%PDF")) {
    reasons.push("pdf_magic_bytes");
    return {
      kind: "brokerage_statement_pdf",
      confidence: "high",
      reasons,
      structureSignature: "pdf:v1",
    };
  }

  if (/<OFX>|<BANKTRANLIST>|<INVTRANLIST>/i.test(textPrefix)) {
    reasons.push("ofx_tags");
    return {
      kind: "ofx_qfx",
      confidence: "high",
      reasons,
      structureSignature: "ofx:v1",
    };
  }

  if (mimeType === "application/json" || textPrefix.trim().startsWith("{")) {
    reasons.push("json_shape");
    return {
      kind: "api_json_export",
      confidence: "medium",
      reasons,
      structureSignature: "json:" + hashKeys(textPrefix),
    };
  }

  if (looksLikeCsv(textPrefix)) {
    const header = firstCsvHeaderLine(textPrefix).toLowerCase();
    reasons.push("csv_delimiter");
    if (header.includes("transaction") || header.includes("amount")) {
      return {
        kind: "transactions_csv",
        confidence: "medium",
        reasons,
        structureSignature: "csv_tx:" + hashHeader(header),
      };
    }
    if (header.includes("symbol") && header.includes("quantity")) {
      return {
        kind: "positions_csv",
        confidence: "medium",
        reasons,
        structureSignature: "csv_pos:" + hashHeader(header),
      };
    }
  }

  return {
    kind: "unknown",
    confidence: "low",
    reasons: ["no_strong_signature"],
    structureSignature: "unknown:v1",
  };
}
```

## Adapter Registry Pattern (API + Upload Unification)

Implement one contract for all ingestion adapters, including APIs and files.

```ts
export interface IngestionAdapter {
  sourceKey: string;
  supports(input: { kind: string; mimeType?: string }): boolean;
  normalize(input: unknown): Promise<{
    extraction: PortsieExtraction;
    observations: Array<{ path: string; value: unknown; confidence: "high" | "medium" | "low" }>;
  }>;
}

export class AdapterRegistry {
  constructor(private adapters: IngestionAdapter[]) {}

  resolve(kind: string): IngestionAdapter {
    const adapter = this.adapters.find((a) => a.supports({ kind }));
    if (!adapter) throw new Error(`No adapter for kind=${kind}`);
    return adapter;
  }
}
```

Benefits:
- APIs and documents are normalized through the same path.
- Every adapter emits canonical extraction + unmapped observations.
- Unknown/new structures become measurable instead of silent failures.

## Rigorous Schema Governance Workflow

### Proposal lifecycle

1. **Observe**
   - collect unknown field paths from `ingestion_observations`.
2. **Aggregate**
   - daily job groups by source + field path + structure signature.
3. **Propose**
   - open/update row in `schema_change_proposals`.
4. **Review**
   - apply checklist before approval:
     - enough evidence count?
     - source-specific or generally canonical?
     - privacy implications?
     - backfill strategy?
     - rollback plan?
5. **Implement**
   - create migration + adapter update + validator update + tests.
6. **Verify**
   - run extraction replay tests for representative samples.

### Hard rules for DB changes

- never add columns to canonical tables for one-off source quirks,
- prefer enum/table-driven lookup over repeated CHECK-constraint edits,
- every migration must be idempotent (`if not exists`, `create or replace`),
- include explicit rollback strategy in migration PR notes,
- require replay tests against historical files before merge.

---

## Recommended Near-Term Refactors

### 1) Replace repeated `data_source` CHECK edits with registry-backed values

Current pattern requires editing constraints across multiple tables for each source addition.
Prefer a normalized source registry and validation function/policy.

Pragmatic intermediate step:
- keep existing checks for safety,
- add `ingestion_sources` now,
- gradually move code paths to source registry lookups,
- remove hardcoded source checks after full migration.

### 2) Add unknown-field capture during validation

Extend `validateExtraction` to emit "accepted unknowns" and "dropped unknowns" as observations.
Persist them to `ingestion_observations`.

### 3) Introduce replay harness for schema change safety

Build a script that replays stored sample extraction payloads through:
- validator,
- account matcher,
- db row builder (dry-run mode),
- integrity checks.

## Implementation Plan (Claude 4.6)

### Phase 1: Foundation (safe, additive)
- add new ingestion/governance tables,
- add detector module for uploads,
- add ingestion run tracking around extract/confirm/verify and API sync routes.

### Phase 2: Adapter unification
- add adapter registry abstraction,
- implement adapters for:
  - current upload extraction output,
  - Schwab API snapshots,
  - Quiltt sync payload mapping.

### Phase 3: Observation + proposal pipeline
- persist unmapped fields,
- add daily proposal generator job,
- add internal admin endpoint/report for open schema proposals.

### Phase 4: Constraint modernization
- progressively reduce hardcoded source enum constraints,
- migrate to source registry-driven governance,
- keep compatibility layer until all routes are switched.

---

## Validation and Testing Requirements

Before production rollout, require:

1. integration tests for mixed-source portfolio assembly (Schwab + Quiltt + uploads),
2. deterministic tests for detector confidence and structure signatures,
3. migration tests for new tables and indexes,
4. replay tests for at least 20 representative uploads and 3 API payload snapshots,
5. regression tests that confirm canonical table shapes remain stable.

---

## Expected Outcomes

If implemented, Portsie gains:

- faster onboarding of new sources with less schema churn,
- safer handling of unknown fields without data loss,
- auditable, disciplined schema evolution,
- consistent ingestion behavior across API and upload channels,
- a compounding learning loop where every new structure improves future ingestion quality.
